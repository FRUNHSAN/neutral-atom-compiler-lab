# 四编译器设计哲学差异

> 不是"谁更好"——是"它们各自在解决什么问题"。
> 基于 224 次独立运行、4 编译器源码通读、12 个 bug 的经验归纳。

---

## 一、每个编译器在管什么（以及不管什么）

### ZAP：只管一件事

**ZAP 唯一真正的决策：空闲比特留还是搬？（Eq.15）**

```
if (1 - f_2q_idle) > (1 - f_tr_move):
    move to storage    # 闲着的代价 > 搬走的代价
else:
    stay in entangling # 搬走的代价 > 闲着的代价
```

这个决策是逐比特、逐阶段、贪心的。不回溯、不搜索、不优化。

ZAP 不管的：
- **门调度**：ASAP（as soon as possible）——纯机械规则，所有能并行的门放在一个 stage。没有"哪种 grouping 更好"的判断。
- **Qubit 放置**：baseline（线性映射）——逻辑 qubit i → 物理 site i。不需要 SA。
- **AOD 路由**：机械的——每个 stage 开始前，把需要的 qubit 搬进纠缠区，结束后把"该搬走"的搬回存储区。
- **Transpilation**：完全交给 Qiskit。

ZAP 的论点：**在中性原子的 zone 架构下，编译器要解决的问题退化到只剩一个——空闲比特管不管。**

---

### ZAC：管所有事，而且每个都要"最优"

ZAC 的设计继承超导编译器传统——placement 和 routing 都是 NP-hard 优化问题：

- **Placement**：SA 迭代。尝试 N 种 qubit→site 映射，每种算一个 cost（传输距离 × 频率），选 cost 最低的。
- **Routing**：SAT-based。把 AOD 移动编码为 SMT 约束（"qubit A 不能同时移动和参与 CZ 门"）；Z3 solver 找可行解。
- **Scheduling**：dependency-aware。不是 ASAP——先建依赖图，确定哪些门必须顺序执行，哪些可以并行。

ZAC 不管的：
- **Crosstalk**：simulator 里的 crosstalk 计算被注释掉了。这行代码的存在证明 ZAC 团队**想过**做 crosstalk，但最终没做。
- **Per-qubit idle decision**：ZAC 没有 Eq.15 等价物。它的 idle qubit 处理是 placement 的副作用（放在纠缠区就暴露，放在存储区就安全），不是独立决策。

ZAC 的隐含假设：**编译质量和搜索空间大小成正比。搜索得越彻底，结果越好。**

---

### PowerMove：管搬运距离

PowerMove 的核心是一个多 AOD 搬运规划器：

- **Graph coloring**（Vizing 边着色）：把 CZ 门分组——同色的门可以用不同 AOD 并行执行。
- **SA placement**：找最小化搬运距离的 qubit→site 映射。
- **Multi-AOD routing**：规划多个 AOD 的移动路径，保证并行无冲突。
- **CZ throttling**（最新 commit）：检测搬运依赖图中的环，把互相卡住的 CZ 门推迟到下一个 stage。

PowerMove 不管的：
- **Idle qubit policy**：没有单独的策略。所有 qubit 都在同一个网格里，没有"存储区"和"纠缠区"的分隔。
- **Decoherence-aware scheduling**：不优化执行时间——优化的是搬运距离。
- **Crosstalk**：完全不考虑。

PowerMove 的隐含假设：**搬运是核心成本。搬得少 = 保真度高。**

---

### Enola：管 layout

Enola 是四者中最老的、最基本的：

- **Layout**：SA 搜索最佳的 qubit→site 初始布局。
- **Routing**："maximalis"——每次移动尽可能多的原子并行。
- **Dependency**：可选的依赖图分析（决定哪些门可以并行）。

Enola 不管的：
- **Zone 架构**：只有一种 zone。qubit 不搬——它们在固定位置，门通过 AOD 把远距离 qubit 搬到一起。
- **Crosstalk**：不考虑。
- **Per-qubit decision**：没有。所有 qubit 一视同仁。

Enola 的隐含假设：**Layout is everything。layout 对了，路由就简单。**

## 二、设计哲学的根本分歧

### 它们来自不同的"前代编译器传统"

| 编译器 | 来源传统 | 核心问题 | 默认工具 |
|---|---|---|---|
| ZAP | 中性原子硬件工程 | 空闲比特管不管 | 贪心决策 |
| ZAC | 超导编译器 (Qiskit Transpiler) | placement + routing 联合优化 | SA + SAT |
| PowerMove | 超导 + 网格 | 最小化搬运距离 | 图着色 + SA |
| Enola | 双原子种硬件 | 初始 layout | SA |

**ZAP 和其他三个的根本分歧不在算法选型——在于对"编译器要解决什么问题"的判断不同。**

- ZAC / PowerMove / Enola 认为：**编译 = 搜索**。有搜索空间（placement, routing, scheduling），有目标函数（距离, 门数），有 solver（SA, SAT, graph coloring）。编译器的工作是在这个空间里找最优点。

- ZAP 认为：**在 zone 架构下，这个搜索空间退化到 trivial**。Placement 不需要搜索（纠缠区内所有 site 等价）；Routing 不需要搜索（只沿一条 1D 轴搬运）；Scheduling 不需要搜索（ASAP 就够）。唯一还有自由度的就是 idle qubit——搬还是不搬。而这个决策是局部的、逐比特的、不需要搜索。

### 用尺子量一量

用我们的框架语言说：

```
ZAP 的 boundary（边界） 只定义了一个 cost：idle vs transfer 的相对代价
ZAC 的 boundary       定义了三个 cost：(1) placement 距离 (2) routing 步数 (3) 验证通过率
PowerMove 的 boundary 定义了三个 cost：(1) 搬运距离 (2) gate 并行度 (3) AOD 冲突
Enola 的 boundary     定义了一个 cost：初始 layout 到最优 layout 的距离
```

**ZAP 的边界是最窄的**——它只承认一个有效的 cost term（Eq.15）。其他编译器把 placement、routing、scheduling 都当成可优化的 cost，而 ZAP 说这些都不需要优化。

## 三、论文里跑不了的东西——精确列表

### 完全不可复现的指标

| 指标 | 原因 | 影响级别 |
|---|---|---|
| PowerMove F_wo_1q (10/14 benchmarks) | Vizing 边着色 bug，不改源码修不了 | 论文的"PM vs ZAP"那 10 个数字无法验证 |
| Enola 全部数据 (14/14) | >180s/benchmark；论文没给环境锁 | 论文的 Enola 柱无法独立复现 |
| ZAC f_idle（crosstalk 层） | 计算被注释掉；论文用后处理脚本填了值 | Fig.7 的绿色柱（串扰）不是 ZAC 原生输出的 |
| QFT N=500 编译时间 | QASM3 transpile O(N²)，60GB 不够 | Fig.12 QFT 面板在一般机器上不可行 |

### 需要额外假设才能比较的指标

| 指标 | 问题 | 需要的假设 |
|---|---|---|
| ZAC F_2q | 二次 transpile 导致门数不同 | 假设 F_2q 差异可忽略（论文 Fig.9 砍掉 F_2q 就是这个原因） |
| 跨编译器"相同 fidelity 公式" | 后处理脚本不在公开仓库 | 假设论文的后处理没有注入 bias |
| qft_n10 F=0.541 | δ=-2.1% | 假设 qiskit 版本差异在可接受范围内 |
| ">1000× speedup" | ZAP 和 ZAC 测量的不是同一段时间（ZAP 不含 transpile，ZAC 含） | 假设 transpile 时间对 speedup ratio 影响 <10% |

### 论文没测但我们能测的

| 指标 | 我们的发现 |
|---|---|
| ZAP scheduler 真正的时间复杂度 | Ising N=500: 30s。O(N) 成立但不是"忽略不计" |
| Cat 电路的 O(N²) 退化 | N≥300: Router stage 迭代开销超线性 |
| hash() 非确定性 | stress test 在所有机器上不可复现（PYTHONHASHSEED） |
| Dynamic vs Always Move 的真正差距 | 默认参数下 ERR≈0.002——策略差异远小于架构差异 |

## 四、四编译器的"选择集"对比

把每个编译器面对同一个电路时必须做的选择列出来：

```
同一种输入: QFT_n10, 10 qubits, 45 CZ gates

ZAP 要做 10 个选择: 每个 qubit 在每个 stage 后搬不搬
    选择空间: 2^(10 qubits × ~7 stages) ≈ 2^70（但 Eq.15 把它坍缩到 1 条路径）

ZAC 要做:
    - placement: 从 N! 种映射中 SA 搜索 ~1000 个候选
    - routing: SAT solver 搜可行解（指数最坏，多项式平均）
    - scheduling: dependency graph 拓扑排序（多项式）
    选择空间: 天文数字（但 SA 只采样 ~1000 个点）

PowerMove 要做:
    - graph coloring: Vizing 算法找最优边着色（多项式，但不保证最优）
    - placement: SA 搜索 N! 种映射
    - routing: 多 AOD 路径规划 + 环检测
    选择空间: 天文数字（但 SA 只采样 ~1000 个点）

Enola 要做:
    - layout: SA 搜索 N! 种初始布局
    - routing: maximalis 搬运规划
    选择空间: 天文数字（SA 采样）
```

**关键**：ZAP 的选择空间大小是 1（Eq.15 是确定性函数，没有随机种子，没有 SA 迭代）。其他三个的选择空间大小是"SA 迭代次数 × 候选数"——取决于运气和初始种子。

这也是为什么我们的 hash 修复对 ZAP 如此重要：ZAP 承诺"确定性"，但 `hash(circuit_type)` 在 Python 3 里就是非确定的——它在 ZAP 唯一的决策点之外引入了一个隐藏的随机源。

## 五、总结

ZAP 在一个拥挤的编译器赛道上做了一个哲学转向：

**"你的优化空间大部分是幻影。在 zone 架构下，placement / routing / scheduling 都不需要优化——只有 idle qubit 策略需要决策。你花在 SA 上的每一秒都在优化一个不存在的自由度。"**

论文的数据支持这个转向，但它只**证明了这个转向对 ZAP 成立**——没有证明其他编译器的优化确实浪费了。

我们无法验证的那部分（ZAC 的 crosstalk、PowerMove 的 10 个 benchmark、Enola 的全部）恰好是这个论证链中最薄弱的环节：**跨编译器"公平对比"依赖一个未公开的后处理层**。

这就像：
- ZAP 说："我的 Eq.15 决策比你的 SA 好。"
- 审稿人问："你怎么知道？"
- ZAP 说："看 Fig.7 的对比。"
- 审稿人问："这个对比是怎么算的？"
- ZAP 说："……有个后处理脚本，用统一的 fidelity 公式重新算了一遍所有编译器的输出。"
- **这个脚本不在公开仓库里。**
